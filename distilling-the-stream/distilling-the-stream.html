Outbox, Inbox & Idempotency

At-least-once delivery is a feature. Duplicates are not.

— Piotr Rybarczyk

The Problem

Same event. Processed twice.

Consumer A
Processing msg #42
✗ Dead
⚡ Broker restart triggers rebalance
Consumer B
Idle
Re-processes msg #42
Message #42 processed twice

This is not a bug. This is Kafka. At-least-once delivery is the contract.

Distilling the Stream

Rule #1: Domain-Level Idempotency First

  • Design business logic so processing the same event twice has no side effect
  • Use natural keys + unique constraints — let the database enforce it
Naive Handler
Message arrives
▼
Side effect executes
▼
Same message arrives again
▼
Side effect executes AGAIN
▼
Duplicate effect
Idempotent Handler
Message arrives
▼
Check natural key / unique constraint
▼
Same message arrives again
▼
Duplicate detected → skip
▼
Same result

Idempotency is a design decision, not a library feature.

Infrastructure patterns help. They do not guarantee it.

The Inbox: Two Strategies

Ecto Strategy (Postgres)

"payments" => %{
  handle_message_plugins: [
    {UUIDIdempotencyPlugin, [
      strategy: EctoInboxStrategy
    ]}
  ]
}

# UUID persisted in partitioned table
# ~2ms overhead per message
# Pruned with PgPartman

Redis Strategy (high-throughput)

"notifications" => %{
  handle_message_plugins: [
    {UUIDIdempotencyPlugin, [
      strategy: RedisInboxStrategy
    ]}
  ]
}

# UUID checked in Redis
# Sub-ms overhead
# TTL-based expiry

Same plugin, swappable strategy. Pick based on your throughput and durability needs.

The Other Side: Transactional Outbox + CDC

# Domain change + event in ONE database transaction
def confirm_appointment(appointment, params) do
  changeset = Appointment.confirm_changeset(appointment, params)

  Multi.new()
  |> Multi.update(:appointment, changeset)
  |> Multi.run(:outbox, fn repo, %{appointment: confirmed} ->
    repo.insert(%OutboxEvent{
      topic_name: "appointments.events-v1",
      event_type: "appointment_confirmed",
      partition_key: confirmed.provider_id,
      payload: Protobuf.encode(confirmed)
    })
  end)
  |> Repo.transaction()
end

Debezium reads PostgreSQL WAL, publishes to Kafka. No polling. Zero coupling.

Your app writes to Postgres. Debezium does the rest — near real-time CDC.

The Real Problem

MSK Restarts: What Actually Happens

  • AWS MSK broker restarts during maintenance windows — routine, expected
  • Kaffe consumers left in wrong state after restart — no visibility into what happened
  • Consumer lag growing silently. No alerts — process was technically "alive" but not consuming
  • KafkaEx + OTP supervision: consumer process crashes cleanly, supervisor restarts it, reconnects automatically, telemetry reports the restart

The difference: Kaffe tried to handle the error. KafkaEx let it crash.

One approach left consumers in limbo. The other recovered in seconds.

Distilling the Stream

MSK Restarts: Let It Crash

# When the broker goes away...
# KafkaEx consumer crashes. Supervisor restarts it.
# It reconnects. No special recovery code.

defmodule MyApp.Application do
  use Application

  def start(_type, _args) do
    children = [
      MyApp.ConsumerSupervisor  # ← restarts on crash
    ]
    Supervisor.start_link(children,
      strategy: :one_for_one)
  end
end

This is not clever engineering. This is OTP doing what OTP does.

Kaffe needed special recovery handlers for MSK restarts. We wrote zero recovery code.

The Payoff: Jumping Two Major Versions

Before: kafka_ex + kayrock on old kpro_schema
  • Produce V0–V5 only
  • No flexible versions (KIP-482)
  • No compact types
  • Stuck on older Kafka clusters
After: updated kpro_schema + generator fixes
  • Produce V0–V8
  • Full KIP-482 support
  • compact_string, compact_array, tagged_fields
  • Works with latest Kafka
What we actually changed:
serialize.ex +60 lines compact_type functions
deserialize.ex +50 lines compact_type parsing
generate.ex +40 lines flexible header logic
record_batch.ex ~0 lines (format unchanged!)
produce.ex regenerated automatic from schema
~150 lines of manual changes
→ hundreds of structs regenerated

The generator design made a 2-version jump tractable.

Distilling the Stream

The Kafka You Deploy Is Changing

  • KRaft mode — ZooKeeper is gone. Simpler operations, faster failovers.
  • Tiered storage (KIP-405) — hot data on disk, cold data in S3. Infinite retention, lower cost.
  • Diskless brokers (WarpStream, AutoMQ) — no local disks at all. Object storage only. Dramatically cheaper MSK alternatives.
  • Shared-nothing rebalancing — consumer groups recover in seconds, not minutes.

The generator design is why we can adopt these changes painlessly.

Same Kafka protocol. Same KafkaEx calls. Same Kafkaesque consumers. Update the schema, regenerate, ship.
